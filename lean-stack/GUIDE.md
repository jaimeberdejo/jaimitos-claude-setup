# The Lean Stack: Complete Setup, Tutorials & Mini-Project

Everything you need to run the lightweight "GSD-without-the-bloat" stack: a ready scaffold, five worked use cases, step-by-step tutorials, and a full mini-project to build from zero.

This guide pairs with the `lean-stack/` folder. Drop that folder's contents into any repo and you have the whole system: auto-maintained docs, deterministic hooks, an independent evaluator, and a guarded autonomous loop.

---

## Part 1 — What you just got (the scaffold)

```
your-repo/
├── CLAUDE.md                      # lean constitution: commands + working agreement
├── .gitignore                     # ignores control/scratch/evidence files
├── .github/workflows/ci.yml       # regression CI: shell syntax, settings, hook smoke tests
├── docs/
│   ├── SPEC.md                    # what & why (fill via the roadmap/grill workflow)
│   ├── ROADMAP.md                 # phases with checkboxes + "Done when" (the roadmap skill writes this)
│   ├── STATE.md                   # where we are + next action (auto-injected)
│   ├── ARCHITECTURE.md            # one-page map, regenerated by the mapme skill
│   ├── plans/                     # per-phase plans /phase writes
│   └── decisions/                 # ADRs, one file each
├── scripts/
│   ├── autopilot.sh               # fresh-context headless loop w/ guardrails (overnight)
│   ├── doctor.sh                  # health check — run before unattended runs
│   └── test-hooks.sh              # smoke-test the hooks
└── .claude/
    ├── settings.json              # wires hooks → events + permissions.deny (secret reads/exfil)
    ├── commands/
    │   ├── resume.md              # /resume    — orient at session start
    │   ├── wrap.md                # /wrap      — close out + update docs
    │   ├── phase.md               # /phase     — build one roadmap phase (no self-tick)
    │   └── autopilot.md           # /autopilot — watchable in-session loop of N phases
    ├── agents/
    │   └── evaluator.md           # independent grader (default-FAIL, no edit tools)
    ├── rules/
    │   └── high-stakes.md         # path-scoped: extra care for auth/migrations/money/etc.
    └── hooks/                     # (+ ownership-nudge.sh reminds you to ADR/teach-back)
        ├── session-start.sh       # re-injects STATE+ROADMAP+findings+git log into context
        ├── format-on-edit.sh      # auto format/lint after every edit
        ├── test-gate.sh           # opt-in deterministic test gate (LEAN_TEST_GATE)
        ├── commit-on-stop.sh      # honest git checkpoint each turn
        ├── kill-switch.sh         # AGENT_STOP blocks the next tool call onward
        └── steer.sh               # STEER.md injects mid-run guidance (JSON additionalContext)
```

> **Where the canonical descriptions live:** the repo-root `README.md` is the master map of
> the whole system; this GUIDE is the long-form manual. If a quick-start snippet ever
> disagrees with these two, these two win — update the snippet.

### What each layer replaces in GSD

| GSD did this automatically | Here it's done by | Token cost |
|---|---|---|
| Generate PROJECT.md / CONTEXT.md | `grill-with-docs` skill | low |
| Generate & tick ROADMAP.md | you (one plan prompt) + `/wrap` | low |
| Persist STATE across resets | `session-start.sh` hook + STATE.md | ~zero |
| Verify each phase | `evaluator` subagent + tests | medium |
| Wave/phase execution | `/phase` + `autopilot.sh` | controlled |
| Context isolation (fresh subagents) | fresh `claude -p` per loop iteration | ~zero overhead |

The difference from GSD: no research fan-out, no 12-agent planning swarm, one context at a time. You trade a little automatic magic for a large token saving (typically several-fold in practice, though it depends on your workload — measure it for yours) and full visibility.

## Part 2 — One-time setup

### Install
```bash
# 1. Skills (Matt Pocock's lean set — grill-me, grill-with-docs, tdd, diagnose)
npx skills@latest add mattpocock/skills

# 2. OpenSpec (optional but recommended for spec-of-record lifecycle)
npm install -g @fission-ai/openspec

# 3. Drop the scaffold into your repo, then:
cd your-repo
git init
openspec init          # if using OpenSpec
chmod +x .claude/hooks/*.sh scripts/*.sh
bash scripts/doctor.sh # health check: tooling, scaffold, settings, hooks
bash scripts/test-hooks.sh   # smoke-test the hooks before trusting them
```

### Secrets: `.gitignore` is not enough
`.gitignore` stops git from *committing* `.env`, but it does nothing to stop Claude
from *reading* it. `.claude/settings.json` ships with a `permissions.deny` block
covering `.env*`, `secrets/**`, `*.pem`, `*.key`, and credential files. Extend it for
your project's sensitive paths.

**Know what's a boundary vs. defense-in-depth:**
- The `Read(...)` deny rules are a real boundary — they block the Read tool on those paths.
- The `Bash(cat *.env*)` / `Bash(curl *)` deny rules are **best-effort, not a boundary** —
  a determined agent could still reach a secret via `less`, `head`, `grep`, `source`, or
  `python -c "open('.env')"`. Enumerating shell commands is whack-a-mole. They raise the
  bar (and block the obvious accidents), nothing more.
- The actual boundary for shell is **sandboxing**: run with sandboxed bash + the 2026
  `sandbox.credentials` setting (blocks sandboxed commands from reading credential files /
  secret env), or run the whole loop in a container with no prod credentials in its env.
  Keep `permission_mode: default` for anything sensitive so you approve shell calls.
- The `Bash(curl *)`/`Bash(wget *)` denies are deliberately conservative; if you do legit
  shell HTTP in your project, remove those two lines.

### Configure your shell (once per machine)
```bash
# export ENABLE_TOOL_SEARCH=1               # UNVERIFIED: not in current Claude Code docs.
                                            # Leave commented unless you confirm it for your version.

# ⚠ DO NOT blanket-export CLAUDE_CODE_SUBAGENT_MODEL=haiku.
# Precedence is: env var > per-invocation > frontmatter. So this env var OVERRIDES
# the evaluator's `model: sonnet` frontmatter — silently downgrading your independent
# grader to haiku, which is the one place you want the stronger model.
# Options that keep the grader strong:
#   (a) leave it unset (subagents inherit the session model), or
#   (b) set it to sonnet, or
#   (c) export it only for cheap helper subagents, never around a /phase grading run.
# The autopilot runs the grader via `claude --agent evaluator` (a main session, not a
# subagent), so that external grade is unaffected — but /phase's in-session evaluator
# subagent IS subject to the env var. Plan accordingly.
```

### Fill in the three placeholders in `CLAUDE.md`
Replace `<NAME>` and the `<...>` command lines with your project's real test/lint/run commands. This file loads every session — keep it under ~500 tokens.

### Set a budget backstop
In your Claude Code config or API gateway, set a hard daily spend cap. The loop guardrails (max-iterations, kill-switch) are your first line; the budget cap is the backstop that saves you from a runaway overnight run.

## Part 3 — The core loop (memorize this)

Everything in this stack is one rhythm:

```
SPEC once  →  ROADMAP once  →  [ /resume → /phase → /wrap → /clear ] × N  →  harden → ship
```

- **SPEC once:** `grill-with-docs` interrogates you, writes `docs/SPEC.md`.
- **ROADMAP once:** the `roadmap` skill breaks the spec into phases with measurable "Done when:" lines.
- **The bracket, repeated per phase:** orient, build one phase (with TDD + evaluator), update docs, clear context.
- **Harden + ship:** full test pass, README, tag.

Manual mode = you drive each arrow. Autopilot mode = `autopilot.sh` drives the bracket for you. Same files, same guardrails.

## Part 4 — Five use cases (worked)

### Use case 1 — A tiny fix (no ceremony)
**"The /health endpoint returns 200 even when the DB is down."**

Don't spin up phases for this. Just:
```
"diagnose: /health returns 200 when the database is unreachable. It should return 503."
```
The `diagnose` skill reproduces → isolates → fixes → adds a regression test. The `format-on-edit` and `commit-on-stop` hooks handle formatting and checkpointing. No SPEC, no ROADMAP, no ADR. **Lesson: match ceremony to stakes.** The stack lets you skip everything for small work — GSD couldn't.

### Use case 2 — A single feature, supervised
**"Add rate limiting to the /transactions endpoint."**
```
1. (plan mode) "grill me on adding rate limiting to /transactions"
       → forces: per-user or per-IP? limit? store? 429 + Retry-After? token bucket vs sliding window?
2. (plan mode) "write the plan to docs/plans/rate-limiting.md"
3. (exit plan) "implement step 1, TDD"
   ... repeat per step ...
4. /wrap   → updates STATE, appends an ADR if you chose a non-obvious store
```
One context, supervised, ~10x cheaper than GSD's equivalent. This is your daily default for medium work.

### Use case 3 — One phase, autonomous
**You have a roadmap; you want phase 2 built without babysitting.**
```
/phase
```
That single command: picks the next unchecked phase, plans it, builds each task TDD, runs the evaluator, and only ticks the roadmap if the evaluator returns PASS. You watch it go and review the diff. If it stalls, it stops and reports rather than thrashing.

### Use case 4 — A whole milestone, overnight
**Build phases 3–6 of a low-stakes project unattended.**
```bash
bash scripts/autopilot.sh 12      # up to 12 iterations
```
Each iteration is a fresh `claude` process (no context rot), grades itself, commits on green. You wake up to a branch of checkpointed commits and an `autopilot.log`. If something went sideways you `git reset` to the last good commit. **Only do this on low-stakes, reversible code** (a static site or internal tool — yes; auth, migrations, or money logic — no).

### Use case 5 — Course-correcting a running loop
**Autopilot is building but you realize it's using `float` for money.**
```bash
echo "Use Python Decimal for all monetary values, never float. Refactor what you've done." > STEER.md
```
The `steer.sh` hook is wired to both `UserPromptSubmit` and `PreToolUse`, so it injects that guidance before the next tool call (mid-run, via JSON `additionalContext`) or at the next prompt — then deletes the file. In a fresh-process `autopilot.sh` loop the guidance lands at the start of the next iteration at the latest. To stop entirely:
```bash
touch AGENT_STOP        # halts all tool calls immediately
# ...fix things...
rm AGENT_STOP           # resume
```

## Part 5 — Tutorials (step by step)

### Tutorial A — From empty repo to first running feature
1. `git init && openspec init`, drop in the scaffold, `chmod +x` the scripts.
2. Edit `CLAUDE.md` placeholders (commands + name).
3. In plan mode: `"grill me on <your project idea>, then write docs/SPEC.md"`.
4. Still in plan mode: `"read docs/SPEC.md, write docs/ROADMAP.md as 4–6 phases, each with Done when"`.
5. Exit plan mode: `/phase`.
6. Review the diff, then `/wrap`, then `/clear`.
7. Repeat 5–6 per phase.

### Tutorial B — Wiring TDD so it's never skipped
The `CLAUDE.md` working agreement asks for TDD, but advisory text is only so reliable. The package ships a deterministic gate: **`.claude/hooks/test-gate.sh`** (wired first in the `Stop` array). It's opt-in via env var so it never surprises you:
```bash
export LEAN_TEST_GATE=warn     # run the suite each turn, write test-results.json, warn on red
export LEAN_TEST_GATE=block    # for money/compliance logic: exit 2 on red — you literally
                               # cannot end a turn with failing tests until you fix them
export LEAN_TEST_CMD="pytest -q tests/unit"   # optional: override the auto-detected command
```
It auto-detects `pytest`/`npm test`, writes `test-results.json` as evidence the evaluator reads, and in `block` mode forces a fix before the turn can end. Leave it `off` (default) for exploratory work; flip to `block` for anything that must not regress.

### Tutorial C — Making the evaluator trustworthy
The evaluator only works if it can't be fooled. Three rules baked into `evaluator.md`: it runs in a **fresh context** (didn't see the build), it has **no Write/Edit tools** (can't "fix" to pass), and it uses a **default-FAIL contract** (every criterion starts false; needs evidence to flip). Never let the builder agent mark its own phase done — always route completion through the evaluator.

### Tutorial D — Persisting context across `/clear`
This is the habit that makes multi-week projects work. The `session-start.sh` hook auto-prints STATE + open roadmap items + recent commits into context on every start/resume/compact. So after a `/clear`, Claude already knows where things stand. You reinforce it with `/resume` when you want an explicit summary before acting. **This single mechanism is the only thing GSD's `.planning/` machinery was really buying you.**

### Tutorial E — Going headless (CI / scheduled)
`autopilot.sh` already uses `claude -p` (headless) and now bakes in the isolation this needs: `bash scripts/autopilot.sh 8 --worktree --pr` runs the whole loop in a fresh `git worktree` on a throwaway branch and opens a PR at the end — it never touches your current branch and never pushes to main. Use `bypassPermissions` only inside a sandboxed container, set the budget cap, and keep `AGENT_STOP` reachable so you can halt a scheduled run. Run `scripts/doctor.sh` first; the bundled `.github/workflows/ci.yml` keeps the hooks themselves from regressing.

## Part 6 — The mini-project: **PrendaPricer**

A small, self-contained, end-to-end project to learn the whole stack on — a throwaway practice project, not tied to anything real. It's a CLI + API that suggests a secondhand-marketplace listing price from an item description, scoped to ~4 phases you can build in an evening.

### Why this project
It exercises every part of the stack: a spec with a measurable success criterion, a roadmap with clean phase boundaries, TDD-friendly pure functions, one external data consideration, and a final autonomous phase you can let run on autopilot. Low stakes (no money moves, fully reversible) so you can safely try `autopilot.sh`.

### The spec (drop into `docs/SPEC.md`)
```md
# Spec: PrendaPricer

## What & why
A CLI + FastAPI service that suggests a listing price for a secondhand
clothing item, given a structured description (category, brand tier, condition,
era/style tags). For my own reselling — faster, more consistent pricing.

## Success criterion (measurable)
Given the 20-item fixture set in tests/fixtures/items.json, the suggested price
is within ±20% of the labelled "good_price" for at least 15 of 20 items.

## In scope
- Pure pricing function: features in → price + confidence out.
- A rules+heuristics baseline (no ML needed for v1).
- CLI (`prendapricer "item desc"`) and POST /price endpoint.

## Non-goals
- No scraping of live marketplace data in v1 (use the fixture set).
- No image input in v1.
- No persistence/database in v1.

## Constraints
- Python 3.12, FastAPI, pytest. Money as Decimal, never float.
- Pricing logic must be a pure, unit-tested function.
```

### The roadmap (drop into `docs/ROADMAP.md`)
```md
# Roadmap

## Phase 1 — Pricing core
- [ ] Define ItemFeatures + PriceSuggestion dataclasses
- [ ] Implement suggest_price() with base-by-category + brand/condition/era multipliers
- [ ] Unit tests for each multiplier and an end-to-end example
Done when: pytest passes and suggest_price() returns a Decimal + confidence for a sample item.

## Phase 2 — Evaluation harness
- [ ] Add tests/fixtures/items.json (20 labelled items)
- [ ] Write an eval test asserting ≥15/20 within ±20% of good_price
Done when: the eval test runs and reports the hit rate.

## Phase 3 — Interfaces
- [ ] CLI entrypoint: prendapricer "<desc>" prints suggestion
- [ ] POST /price FastAPI endpoint returning JSON
- [ ] Endpoint test with TestClient
Done when: curl to /price returns a valid suggestion and the CLI works.

## Phase 4 — Hardening (autopilot-friendly)
- [ ] Input validation + 422 on bad input
- [ ] README with usage + the eval hit rate
- [ ] Tune multipliers until the eval test passes if it isn't already
Done when: full suite green, eval criterion met, README written.
```

### How to build it (the session script)

**Session 1 — scaffold + Phase 1**
```
(plan mode) "/resume"                         # reads SPEC + ROADMAP
(plan mode) "plan phase 1 into docs/plans/phase-1.md"
(exit plan) "implement phase 1, TDD, one task at a time"
# review diff
"@evaluator grade phase 1"
/wrap
/clear
```

**Session 2 — Phase 2 (the eval harness is the heart)**
```
/resume
"/phase"            # let it run phase 2 autonomously; it builds the fixtures + eval test
# review: does the eval test actually assert the ±20% / 15-of-20 criterion?
/wrap
/clear
```

**Session 3 — Phase 3 interfaces**
```
/resume
"/phase"
# manually curl the endpoint + run the CLI to sanity check
/wrap
/clear
```

**Session 4 — Phase 4 on autopilot**
```bash
# Phase 4 is mechanical (validation, README, tuning to pass the eval) — ideal for autopilot.
bash scripts/autopilot.sh 6
# watch autopilot.log; if it tunes multipliers in a way you dislike:
echo "Don't overfit multipliers to the fixtures; keep them interpretable." > STEER.md
```
When `autopilot.sh` reports the roadmap complete and the evaluator PASSes, you have a working, tested, documented tool — built across four sessions, with a full git checkpoint history, ADRs for any real decisions, and a STATE.md you could hand to a stranger.

### What you'll have learned
- Writing a spec with a **measurable** success criterion (the thing that makes autonomy safe).
- Phase boundaries that each leave a working program.
- TDD as the loop's truth source.
- The evaluator catching premature "done."
- Letting a low-stakes mechanical phase run unattended — and steering/stopping it.

### Graduating to high-stakes work
Once the practice project feels natural, apply the same stack to real work with one change for anything consequential: **drop autopilot for high-stakes or hard-to-reverse code** (auth, migrations, money, deletes, external side effects). Use `/phase` supervised, keep `permission_mode: default` on that code, require your review before merge, and let the git history + ADRs be your audit trail. The loop *surfaces* those diffs; you approve them — the human-on-the-loop posture. The `.claude/rules/high-stakes.md` rule encodes this; point its `paths:` at your sensitive directories.

### One paragraph to remember
Native Claude Code plus five small skills, a `docs/` folder, and five short hooks reproduce everything GSD automated — spec, roadmap, persistent state, decision log, phase execution, verification — at a fraction of the tokens, because there's no research fan-out and one fresh context per phase. You drive the loop manually for stakes that warrant it and hand it to `autopilot.sh` when they don't, with a kill-switch, an independent grader, and a budget cap as your safety net. Learn it on the practice project; run real work on it — manually for the high-stakes parts, autonomously for the rest.
