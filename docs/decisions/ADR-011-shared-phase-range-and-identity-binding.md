# ADR-011: One shared phase-range resolver, and grade/evidence bound to phase identity

Date: 2026-07-20
Decision: The precedence that resolves a phase's `BASE..HEAD` window (`TICK_BASE` orchestrator-trusted → tracked `.claude/.phase-anchor` → `.claude/.phase-base`), plus the strict-ancestor guard and the anchor base-integrity check, live ONCE in `.claude/lib/_phase-range.sh` (with a read-only CLI `scripts/phase-range.sh`). `tick.sh` delegates its base resolution to it; the independent evaluator resolves its review window from the same CLI instead of reading `.phase-base` directly; and headless `autopilot.sh` exports the trusted base as `TICK_BASE` for the evaluator too. On top of that, `record-grade.sh` and `test-evidence.sh` stamp the phase `heading` + `base` (evidence `schema_version` 3), and `tick.sh` refuses unless the grade's and evidence's heading+base match the phase being ticked and the resolved base (and verifies the evidence `content_hash`).
Why: The independent review diffed `.claude/.phase-base` while `tick.sh` preferred `.claude/.phase-anchor`, so the review, the evidence, the secret/high-stakes scan and the tick could each judge a DIFFERENT window; and the grade/evidence were bound only to HEAD, so at one commit a grade/evidence recorded for phase X could tick phase Y. One resolver makes every consumer judge the identical range; the identity binding closes the same-HEAD cross-phase reuse. The rejected alternative — keeping per-consumer base logic and adding compensating checks — is exactly the duplicated-canonical-state and fix-in-the-wrong-layer trap the toolkit rules out; the resolver REPLACES tick's inline copy rather than adding a second one. Honest scope is unchanged: manual mode stays tamper-evident, not builder-proof — headless `TICK_BASE` remains the trust-equivalent path.
