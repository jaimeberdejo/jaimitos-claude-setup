# ADR-012: Sync reconciles retired managed files, report-first and opt-in to delete

Date: 2026-07-20
Decision: `sync.sh` reconciles manifest entries the current toolkit no longer ships ("retired" paths). It reports them by default and NEVER deletes without `--prune` plus a confirmation (or `--yes`); it removes only an UNCHANGED retired file (local sha == the manifest sha, inside a managed root, not a symlink), dropping its manifest entry; a LOCALLY-MODIFIED retired file is never auto-removed (reported for a manual decision); a LOCALLY-DELETED one just has its stale manifest entry dropped. Every retired path is path-safety-checked first (no absolute path, no `..` traversal, no symlink, inside `scripts/`/`.claude/`/`sandbox/`/`.github/`).
Why: Before this, sync only classified files the current toolkit still ships, so an upgrade that retired a guard script left it (and its stale manifest entry) on disk forever — the manifest silently disagreed with reality. Reconciling it keeps the manifest honest. But deleting a user's file is the most destructive thing an updater can do, so it is report-first, opt-in, confirmed, and refuses anything it is not certain is a pristine toolkit copy — a modified or unrecognised path is preserved and surfaced, never removed. The rejected alternative — auto-deleting retired files on every sync — trades a silent-residue problem for a silent-data-loss problem, which the toolkit's fail-safe posture forbids.
